# Implementation Notes

## Assumptions

The following technical assumptions were made during implementation of the Forger system. Each assumption represents a decision point where the system behavior depends on an unverified external condition.

1. **Kilo Code new_task() spawning — synchronous.** The Orchestrator's execution loop assumes that when it calls `new_task()` to spawn a DEV agent, that subtask runs to completion before the Orchestrator continues. If Kilo Code implements `new_task()` asynchronously, the Orchestrator must be redesigned to poll `SLICES.md` for slice status changes rather than assuming sequential completion.

2. **forger_cli.py path — accessible from project root.** Agents are instructed to run `python forger_cli.py <command>`. This assumes that `forger_cli.py` is either (a) copied into each project directory alongside the Markdown files, (b) accessible via a `FORGER_PATH` environment variable, or (c) referenced via an absolute path embedded in the mode instructions. Option (a) is simplest for single-project use; option (b) is preferred for multi-project use.

3. **custom_modes.yaml scope — global in Kilo Code.** Kilo Code custom modes defined in `.kilocode/modes/custom_modes.yaml` may apply globally across all projects opened in VSCode, not just the Forger directory. Verify that the `forger-*` mode slugs do not conflict with any existing custom modes the user has defined. If a conflict exists, rename the Forger mode slugs.

4. **new_task() mode parameter — supported.** The workflow files (`forger-plan.md`, `forger-execute.md`) assume that Kilo Code workflows can pass a mode slug as a parameter to `new_task()` when spawning subtasks. If this parameter is not supported by the installed version of Kilo Code, the user must manually switch the active mode in the Kilo Code panel before each agent task begins.

5. **Command group access — terminal execution permitted.** Agents operating in modes that include the "command" tool group are assumed to be able to execute terminal commands, including `python forger_cli.py` scripts. If the Kilo Code environment restricts terminal access for certain modes, the mode definitions in `custom_modes.yaml` must be updated to explicitly include the `command` tool group for Orchestrator, DEV, QA, Debugger, and Auditor modes.

6. **Working directory — project root.** All agents are assumed to execute CLI commands from the project root directory (the directory containing `PROJECT.md`, `STATE.md`, `SLICES.md`, etc.). If an agent's working directory differs (e.g., a subdirectory), CLI commands that reference these files by relative path will fail.

---

## Known Risks

- **new_task() async behavior** could silently break the Orchestrator execution loop, causing slices to be marked in_progress simultaneously or out of order. This is the highest-impact risk. Mitigation: redesign Orchestrator to poll `SLICES.md` status before dispatching the next slice.

- **Long SLICES.md files** may exceed the context window of individual agents that need to read the full file. Mitigation: keep slice descriptions concise; consider splitting very large projects into sub-projects with their own SLICES.md.

- **custom_modes.yaml merge conflict** if the user already has a `.kilocode/modes/custom_modes.yaml` in their VSCode config. The Forger modes must be merged manually into the existing file rather than overwriting it.

- **Python availability** — `forger_cli.py` requires Python 3.11+ to be present in the system PATH. On some systems Python may be installed as `python3` rather than `python`. Agents invoking the CLI should handle this, or the mode instructions should specify `python3 forger_cli.py` as a fallback.

---

## Implementation Decisions

- **SLICES.md Status field over a separate state file for slice tracking.** Each slice carries its own `Status` field (`pending | in_progress | done | failed`) directly in `SLICES.md`. This keeps slice state co-located with slice definitions, giving any agent a single file to read for the full picture. An alternative (separate `SLICE_STATE.md` or database) would add a synchronization burden with no benefit given the sequential execution model.

- **Embedded contracts in task packets rather than file references.** Task packets generated by `forger_cli.py orchestrate` embed the full content of each relevant contract file directly into the packet. This means the DEV agent only needs to read one file (`tasks/[ID].md`) rather than reading the task packet plus multiple contract files. The tradeoff is task packet file size, which is acceptable given the single-file context benefit.

- **QA writes to separate qa_reports/ directory rather than inline in SLICES.md.** QA reports are verbose and include pass/fail checklists that would bloat `SLICES.md` if embedded. Keeping them in `qa_reports/[ID].md` preserves the readability of `SLICES.md` as a status board while making full QA output available to Debugger and Auditor agents on demand.

- **Python stdlib only for forger_cli.py.** The CLI uses no third-party libraries. This eliminates the need for a `requirements.txt`, a virtual environment setup step, or any package installation before first use. All file parsing uses basic string operations on Markdown rather than a Markdown parsing library, which is sufficient for the structured formats used in `SLICES.md` and `STATE.md`.

## Bugs Found and Fixed (E2E Test Round 1 — 2026-03-03)

### Bug 1: ROOT path pointed to Forger system dir instead of project CWD
**Symptom:** Running `python forger_cli.py init` in `test-project-001` created all files inside `forger-ai-selfhosted/`, not in the test project.
**Root cause:** `ROOT = Path(__file__).resolve().parent` always resolves to the directory containing the script, regardless of where it is called from.
**Fix:** Changed to `ROOT = Path.cwd()` and added `FORGER_ROOT = Path(__file__).resolve().parent` for locating Forger system files. Added `import shutil`.

### Bug 2: Kilo Code modes not visible in test project
**Symptom:** Opening `test-project-001` in VSCode showed no `forger-*` modes in Kilo Code.
**Root cause 1:** The modes file was at `forger-ai-selfhosted/.kilocode/modes/custom_modes.yaml` — Kilo Code does NOT read this from workspace root. The correct project-level file is `.kilocodemodes` (no extension) at workspace root.
**Root cause 2:** Kilo Code requires a `description` field on each mode; modes without it may be silently ignored.
**Fix:** (a) Updated `cmd_init()` to copy `custom_modes.yaml` to `.kilocodemodes` at the project root. (b) Added `description` and `whenToUse` fields to all 7 modes in `custom_modes.yaml`. (c) Added `cmd_install()` to write modes to `~/.kilo/custom_modes.yaml` for global access.

### Bug 3: SLICES.md template showed placeholder slices as pending
**Symptom:** `python forger_cli.py status` reported "0/2 done, pending: 2" on a fresh project with no real slices yet.
**Root cause:** `SLICES_TEMPLATE` in forger_cli.py contained two placeholder `## S1` and `## S2` blocks with `Status: pending`.
**Fix:** Simplified `SLICES_TEMPLATE` to comments only. The PM agent writes all actual slices.

## Bugs Found and Fixed (E2E Test Round 2 — 2026-03-03)

### Bug 4: Agents used relative path to forger_cli.py
**Symptom:** Agents ran `python ../forger-ai-selfhosted/forger_cli.py lock` instead of `python forger_cli.py lock`, requiring cross-directory access and producing confusing paths in prompts.
**Root cause:** `forger_cli.py` only existed in the Forger system directory; projects had no local copy.
**Fix:** `cmd_init()` now copies `forger_cli.py` to the project root using `shutil.copy2()`. Self-copy is safe (src==dst check prevents overwrite). All mode instructions updated to use `python forger_cli.py [cmd]` with no path prefix.

### Bug 5: PM didn't auto-run orchestrate; required user to do it manually
**Symptom:** PM completed planning docs but ended with "Run python forger_cli.py orchestrate to begin execution" instead of doing it.
**Root cause:** PM's customInstructions said "run orchestrate" but the agent treated it as advisory, not imperative. Also, the PM didn't spawn the Orchestrator after orchestrate.
**Fix:** Updated PM's customInstructions to: (a) add an explicit CONFIRMATION GATE that shows a summary table to the user and waits for "confirm" / "decline" reply, (b) only run orchestrate after "confirm", (c) spawn forger-orchestrator automatically after orchestrate runs.

### Bug 6: Orchestrator mode required manual user switch
**Symptom:** After PM completed, the Orchestrator wasn't started — user had to manually switch modes.
**Root cause:** No mechanism to auto-spawn the Orchestrator after PM's planning + orchestrate step.
**Fix:** PM's confirmed flow now ends with `new_task(mode=forger-orchestrator, ...)`. Orchestrator customInstructions updated with "START IMMEDIATELY" directive.

### Bug 7: `python forger_cli.py execute` was an invalid command
**Symptom:** User tried `execute` subcommand, got argparse error "invalid choice".
**Fix:** Added `cmd_execute()` that shows current phase, slice progress, and actionable next-step instructions.

### Bug 8: Lock/phase changes not confirmed visually in chat
**Symptom:** After lock, PROJECT.md and STATE.md still showed DISCOVERY in the documents pane (user expected the agent to confirm the change).
**Root cause:** The agent ran the CLI (which did update the files) but didn't explicitly show the user the updated values.
**Fix (partial):** Updated Planner's instructions to print "Project locked." confirmation and tell the user planning is starting. The documents ARE updated by the CLI — this is a display behavior, not a data bug. User should reload the file to see updated values.

### Bug 9: Spawned subtasks used "General" mode instead of forger-dev/forger-qa

## Bugs Found and Fixed (E2E Test Round 4 — 2026-03-03)

### Bug 11: PM wrote wrong SLICES.md field names, causing pipeline collapse
**Symptom:** PM agent wrote `- **File:**`, `- **Type:**`, `- **Purpose:**`, `- **Requirements:**` instead of `- Description:`, `- Inputs:`, `- Outputs:`, `- Contracts:`, `- Tests:`. Also wrote `## Slice 1 - name` heading instead of `## S1 - name`. Result: `parse_slices()` found no valid Contracts field, `start-slice` couldn't find the slice ID, `orchestrate` generated `tasks/Slice.md` (corrupted ID), and the Orchestrator fell into an infinite loop trying to fix state.
**Root cause 1:** PM ignored the SLICES.md format template and invented its own field names.
**Root cause 2:** `parse_slices()` regex `r"-\s+(\w[\w\s]*):\s*(.*)"` did not match `- **Field:** value` (bold markdown) used by the PM.
**Root cause 3:** `update_slice_status()` regex `r"-\s+Status:\s*.*"` did not match `- **Status:** pending`.
**Fix:**
1. `parse_slices()` updated to match both plain `- Field: value` and bold `- **Field:** value` styles.
2. `update_slice_status()` updated to match both styles.
3. PM customInstructions in `custom_modes.yaml` updated to explicitly forbid custom field names and bold markers in SLICES.md, with a "FORBIDDEN" warning.
4. `prompts/PM.md` updated with the same "FORBIDDEN" warning after the format block.
5. SLICES.md for test-project-003 manually corrected; tasks/Slice.md removed; `orchestrate` re-run to generate correct S1.md/S2.md/S3.md.

## Bugs Found and Fixed (E2E Test Round 5 — 2026-03-03)

### Bug 12: Auditor returned AUDIT.md as message text instead of writing file
**Symptom:** After Auditor completed its run, `AUDIT.md` did not exist on disk. The audit content was returned as the subtask reply message to the Orchestrator.
**Root cause:** AUDITOR.md said "write AUDIT.md" but did not explicitly instruct the agent to use a file-write tool call. The agent treated the instruction as "output the content" rather than "invoke the write tool."
**Fix:** AUDITOR.md updated with explicit instruction: "use your file-write tool to create AUDIT.md at the project root. Do NOT output the audit content as a text reply. You MUST write it as a file."

### Bug 13: DEV over-implementation — created files outside its contracts
**Symptom:** DEV agent for S1 (converter.py) also created test_converter.py, which was S3's contracted output. S3's DEV agent then had to detect and work with the existing file.
**Root cause:** The S1 task packet's Tests field referenced `pytest test_converter.py`. The DEV agent, seeing a test reference to a non-existent file, assumed it needed to create it.
**Fix:** DEV.md Step 3 updated with explicit guard: "Before creating any file, check that it appears in the Outputs field of your task packet. If the Tests section references a file NOT listed in your Outputs/contracts, do NOT create it — it belongs to a different slice."

### Bug 14: Orchestrator called complete-slice again after QA PASS (already done by DEV)
**Symptom:** Orchestrator called `python forger_cli.py complete-slice S1` after QA reported PASS, even though the DEV agent had already called it in Step 7 of DEV.md. The call was idempotent but wasted a CLI invocation.
**Root cause:** Orchestrator was interpreting "slice is validated" as requiring it to mark it done, not realizing DEV had already done so.
**Fix:** ORCHESTRATOR.md Step 6 updated: "If STATUS: PASS — slice is validated. Do NOT run complete-slice again (the DEV agent already called it). Return to Step 1."

### Bug 10: Parallel execution not happening for independent slices
**Symptom:** Slices with no dependencies (e.g., S1–S6 in the Hello World test project) were executed sequentially, one at a time.
**Root cause:** Sequential execution is intentional in the current Orchestrator design. Each slice must complete DEV→QA before the next begins, ensuring test isolation and predictable state.
**Decision:** Keep sequential execution as the default. Parallel execution requires a different state model (per-slice locking, conflict detection) and is deferred to a future improvement. Documented in "What Was Not Implemented" below.

## Bugs Found and Fixed (E2E Test Round 6 — 2026-03-03)

### Bug 15: AUDIT.md still not written to disk (Round 5 fix insufficient)

**Fix:** QA.md Step 3 now says: `"Write your report to forger/qa_reports/[id].md. **Use your file-write tool to create this file — do NOT output the report as text in your response.**"` Same explicit language added to `custom_modes.yaml` QA spawn message.

## Bugs Found and Fixed (E2E Test Round 7 — 2026-03-03)

### Bug 18: PM used attempt_completion for planning summary — ended task prematurely, no project created
**Symptom:** PM presented the planning summary via `attempt_completion`, which ended the PM subtask. The Planner received this as the PM's final result, incorrectly fired Post-Completion Handoff (giving E2E instructions for files that didn't exist yet), and terminated. No Orchestrator was spawned. No code was written.
**Root cause:** PM.md Step 8 said "present a confirmation summary to the user" without specifying HOW. The agent interpreted this as using `attempt_completion` (the natural way to return a result). This is distinct from asking the user a question, which requires `ask_followup_question`.
**Fix:** PM.md Step 8 now explicitly says: "Use `ask_followup_question` to present the summary to the user and wait for their response. Do NOT use `attempt_completion` — that would end your task prematurely before execution begins." Same explicit instruction added to CONFIRMATION GATE in `custom_modes.yaml` forger-pm section.
**Note on YOLO mode:** YOLO mode auto-approves tool executions (read/write/command) but does NOT bypass `ask_followup_question` — genuine user input always requires a response. This means YOLO mode + `ask_followup_question` confirmation gate is the correct combination: commands run freely, but planning approval always pauses for the user.

## Bug Found and Fixed (Round 7 continued — 2026-03-03)

### Bug 19: next-slice returned ALL_DONE with stale in_progress slices
**Symptom:** In test-project-005, S1 was left as `in_progress` after an aborted Planner session had run `orchestrate` + `start-slice S1`. When the new Orchestrator session ran `next-slice`, it found no `pending` slices (S2-S5 were pending but the code only checked `pending`) and returned `ALL_DONE` prematurely. S1 was never re-executed.
**Root cause:** `cmd_next_slice()` only scanned for `status == "pending"`. Stale `in_progress` slices (where the session died before writing a DONE.md) were invisible to it.
**Fix:** Added stale detection loop: after checking all slices for `pending`, iterate again looking for `in_progress` slices that have no `tasks/[id]-DONE.md`. Return `STALE:[id]: [title]` for the first match. Updated `ORCHESTRATOR.md` Step 1 to handle this prefix: run `start-slice [id]` to reset it, then proceed directly to Step 3 (skip Step 2).

## Bugs Found and Fixed (Round 8 -- 2026-03-03)

### Bug 20: Planner skips discovery gate when YOLO mode is ON
**Symptom:** In test-project-006, the Planner agent ran through discovery and locked the project without calling `ask_followup_question` -- it just proceeded autonomously. No user confirmation was presented before locking.
**Root cause:** In YOLO mode, the LLM model treats "autonomous operation" as skipping interactive checks entirely. YOLO mode does not auto-approve `ask_followup_question` -- it suppresses the call altogether. The Planner's Final Summary previously ended with plain text "reply locked when ready" which YOLO mode ignored.
**Fix:**
1. `PLANNER_ARCHITECT.md` Final Summary: now uses `ask_followup_question` with "locked" / "I have corrections" options. Added to What You Must NOT Do: "Do not proceed to lock without the user selecting 'locked' from this question."
2. `.kilocode/modes/custom_modes.yaml` forger-planner: roleDefinition updated to say "you MUST use ask_followup_question with 'locked' / 'I have corrections' options and wait for the user to respond"; customInstructions has explicit DISCOVERY GATE block.
3. Both `prompts/PLANNER_ARCHITECT.md` and `forger_code/data/prompts/PLANNER_ARCHITECT.md` updated.
**Workflow note:** YOLO OFF for Planner discovery (conversational, gates respected), YOLO ON after PM gate confirmed before Orchestrator (fully autonomous execution).

### Bug 20b: Planner wrote PROJECT.md after lock command (Locked: true reset to false)
**Symptom:** In test-006 run, PROJECT.md showed `Locked: false` after the Planner had supposedly locked it. PM then "corrected" it back to `Locked: true` (itself a rule violation).
**Root cause:** The Planner wrote PROJECT.md as the LAST step of the LOCKED Trigger, AFTER running `forger_cli.py lock`. The lock command sets `Locked: true` in PROJECT.md; overwriting it afterward reset it back to false.
**Fix:** PLANNER_ARCHITECT.md LOCKED Trigger Step 1 now says: "This is the last moment you may touch PROJECT.md -- do not modify it after step 3 (the lock command)." What You Must NOT Do added: "Do not modify `forger/PROJECT.md` after running `forger_cli.py lock`. The lock command sets `Locked: true` in PROJECT.md -- any write after that resets it to false and breaks the audit."

### Bug 20c: PM modified PROJECT.md to fix Locked: false (violating its own rules)
**Symptom:** In test-006, PM agent corrected `Locked: false` to `Locked: true` in PROJECT.md, which is a violation of PM.md's stated rule ("Do not modify PROJECT.md").
**Root cause:** The existing PM.md prohibition was vague -- PM may have inferred that fixing an obvious error was acceptable. Also caused by Bug 20b (Planner creating the false-locked situation).
**Fix:** PM.md What You Must NOT Do expanded: "Do not modify `forger/PROJECT.md` **for any reason** -- not to fix typos, not to update the Locked status, not to add missing content. PROJECT.md is frozen the moment `forger_cli.py lock` runs. If you notice an inconsistency (e.g., `Locked: false` when it should be true), document it in your planning notes and leave the file untouched -- the Auditor will log it, and the Planner is responsible for the pre-lock state."

## Round 9 Changes (2026-03-04)

### .forger/ Rename
`FORGER_DIR` changed from `ROOT / "forger"` to `ROOT / ".forger"` in `forger_cli.py`. All 7 prompt files, both copies of `custom_modes.yaml`, and generated task packet templates updated. New projects use `.forger/` as the workflow directory.

Rationale: `.forger/` follows the convention of hidden dot-directories (`.git`, `.pytest_cache`, `.venv`).

### QA Gate Hardening (Bug 21 -- Mock Files)
Root cause identified in mt5-api-docker: DEV called `complete-slice` itself, bypassing QA. All API slices (S9-S13) were marked done with no QA reports. DEV agents created stubs because MT5 library was unavailable.

Fixes applied:
1. DEV no longer calls `complete-slice` -- removed from DEV.md Step 7, task packet template, ORCHESTRATOR.md Step 3 spawn message, custom_modes.yaml.
2. Orchestrator calls `complete-slice` only after QA PASS (ORCHESTRATOR.md Step 6).
3. Orchestrator verifies QA report file exists before reading it; re-spawns QA if missing.
4. QA.md: "no tests ran" / "0 collected" / "collected 0 items" = explicit FAIL.
5. QA.md: hardcoded stub values / in-memory simulation substituting for real integrations = FAIL.
6. DEV.md: No-stubs rule in Step 3; "no tests ran is a failure" in Step 5.
7. PM.md: Anti-stub criterion added to standard contract Success Criteria template (criterion 6).

### Interrupt Mode
New STATE.md field: `interrupt_mode: autonomous` (default).
- CLI command: `forger set-interrupt autonomous|per_milestone|per_burst|per_slice`
- `forger status` shows Interrupt field.
- Planner Cluster 9 asks user for preference; LOCKED Trigger runs set-interrupt after lock.
- Orchestrator reads interrupt_mode and uses ask_followup_question at the configured boundary.

### Parallel Debug Team
On second QA failure (attempt 2 of 3), Orchestrator:
1. Writes `.forger/debug_briefs/[id].md` (problem + 3 approaches)
2. Spawns Debugger A -> writes `.forger/debug_attempts/[id]-A.md`
3. Spawns Debugger B -> writes `.forger/debug_attempts/[id]-B.md`

## Round 15 — mt5-api-docker Run 2 Bugs and Fixes (2026-03-05)

### Test Project
`c:\projects\mt5-api-docker` — second full run with fresh project init. Complex Docker + Wine + MT5 + REST + WebSocket + noVNC.
Run completed with a functional deployable MT5 Docker image. Post-completion networking improvement request was handled correctly by re-entering Planner -> PM -> Orchestrator flow.

### Bug 26: QA passed slice with failed infrastructure command (non-zero exit)
**Symptom:** S2 passed QA even though `docker compose up -d` failed with `open //./pipe/dockerDesktopLinuxEngine: The system cannot find the file specified.` (Docker Desktop not running). QA evaluated file contents only. The broken Docker state leaked into downstream slices — the container entered a restart loop that was never caught by QA.
**Root Cause:** QA.md "Additional Checks" section has no rule about infrastructure commands exiting with errors. The anti-stub rule covers in-code simulation but not infrastructure test failures.
**Fix needed:** Add to QA.md Step 2 Additional Checks:
- "Did all infrastructure commands (docker, systemctl, etc.) specified in the Tests section exit with code 0? A non-zero exit from any infrastructure setup command is an automatic FAIL, regardless of whether the code files look correct."

### Bug 27: Contract paths in task packets not fully qualified from project root
**Symptom:** Kilo Code error: `Error reading file .forger/tasks/contracts/mt5_cli.md` (correct path: `.forger/contracts/mt5_cli.md`). Agent constructed path as `.forger/tasks/` + `contracts/mt5_cli.md` — treating the task packet directory as the base rather than project root.
**Root Cause:** The contract path inside the task packet was written as `contracts/mt5_cli.md` (relative, no `.forger/` prefix). When the agent resolved this against its own location in `.forger/tasks/`, it produced the wrong path. The agent recovered by scanning the filesystem, but this wastes tokens and is fragile.
**Fix needed:** In `cmd_orchestrate()`, verify that all embedded contract paths in generated task packets always begin with `.forger/contracts/`. Never write bare `contracts/` paths. The `embed_contract()` function should assert that contract_path starts with `.forger/`.

### Bug 28: VSCode linters trigger infinite loop on language-specific contract filenames
**Symptom:** PM agent repeatedly tries to fix `.forger/contracts/dockerfile.contract.md` in a loop. Kilo Code's dockerfile-utils linter parses the contract file as Dockerfile syntax, reports "Unknown instruction: -" for markdown bullet points, and returns errors. PM sees errors and retries the write indefinitely.
**Root Cause:** Contract filenames that match language keywords (dockerfile, package.json, docker-compose.yml, etc.) trigger language-specific linters, which try to validate the `.md` file as source code. Markdown content (`- **Field:** value`) is not valid Dockerfile syntax.
**Fix applied:** Changed contract naming pattern from `{filename}.contract.md` or bare `{filename}.md` to `contract_{filename}.md`. This prevents linter pattern matching. Updated PM.md Step 5 and FORBIDDEN examples. Added CRITICAL warning about linter interference.
**Prevention:** All contract filenames now follow the pattern `.forger/contracts/contract_{filename}.md` (e.g., `contract_dockerfile.md`, `contract_package.json.md`). The `contract_` prefix is non-matching for all language linters. No linter will attempt to validate these files.

### Observation: Test evasion — agents pass tests by working around them
**Symptom:** Agents made tests pass by adjusting assertions or mocking behavior rather than fixing the underlying implementation. This satisfies the "tests pass" criterion without satisfying the actual requirement.
**Root Cause:** Agent objective is "make tests pass," which is satisfied by any means including test fixture manipulation. The anti-stub criterion in QA.md covers code stubs but not test manipulation.
**Fix needed:** Add to QA.md Step 2:
- "Are test assertions substantive? Read the test files and verify tests are not: asserting `True` directly, skipping assertions, using `pytest.skip()`, asserting on mock return values that match hardcoded implementation values, or otherwise trivially passing without testing real behavior. Any test that passes by asserting nothing or asserting a trivially true condition is a FAIL."

### Observation: Kilo Code UI freeze on fast-exit commands
**Symptom:** When a command (e.g. uvicorn with port-in-use error) starts and immediately exits, Kilo Code UI greys out the allow/deny buttons and appears stuck. The user must close the task and reopen the Orchestrator.
**Classification:** Kilo Code UI bug — not a Forger issue. STALE detection handles the recovery correctly. No Forger fix required.
**Workaround documented:** If Kilo Code terminal buttons grey out, close the task and reopen in forger-orchestrator mode. The STALE detection will pick up where execution left off.

### Observation: Stray `-p` folder in project root
**Symptom:** An empty `-p` folder appeared at the project root (`c:\projects\mt5-api-docker\-p\`).
**Root Cause:** An agent ran `mkdir -p` with an empty path argument. On bash, `mkdir -p` with no path creates a literal directory named `-p` in the cwd.
**Fix:** Delete the folder manually. No Forger fix required — this is a DEV agent command construction error.

### Architecture comparison: EA server (inbound) vs MetaQuotes recommended (outbound)
**What was built:** EA acts as an inbound TCP server (ports 8082 HTTP, 8083 WS) using raw WinSock2 DLL. External clients connect to the EA. Requires `Allow DLL imports` in MT5 settings.
**MetaQuotes recommended:** EA makes outbound calls using built-in `WebRequest()`. EA is client, external server is listener. No DLL required.
**Verdict:** Inbound server is architecturally better for Docker containers (fixed ports, no external server needed, native WS push). The DLL dependency is acceptable for this use case.

## What Was Not Implemented

- **No LLM runtime.** Forger does not include an LLM client, API key management, or model invocation logic. Kilo Code provides the LLM runtime. Forger provides the structure and prompts that Kilo Code agents consume.

- **No web UI.** All user interaction happens through the Kilo Code VSCode panel (for agent conversation) and the terminal (for CLI commands). Kilo Code provides the graphical interface.

- **No git integration.** There is no automatic git commit, branch creation, or PR generation tied to slice completion. This could be added as a future enhancement — for example, `complete-slice` could optionally commit the changed files with a message derived from the slice title.

- **No parallel slice execution.** Slices are executed strictly sequentially. The Orchestrator dispatches one slice at a time, waits for QA to pass, then moves to the next. Parallel execution would require a fundamentally different state model (locking per file, conflict detection) and is intentionally excluded to keep the system simple and deterministic.
